JVM 调优的面试题，很多人的真实反应是：背了几个参数，面试官一问就露馅。

不是因为参数没记住，而是不知道面试官真正想问什么。今天把这层窗户纸捅破。

## 面试官问 JVM 调优，到底在问什么？

面试官问"你说说 JVM 调优"，他想要的不是你把参数列表背一遍。

他想知道的是：

- 你遇到过高并发场景下的性能问题吗？
- 你知道怎么定位问题吗？
- 你会用工具分析问题，还是全靠猜？

**调优不是表演参数记忆术，是你面对一台出问题的服务器，知道从哪里入手。**

我在前公司接手过一个线上服务，高峰时段响应越来越慢，最后直接 OOM 挂了。重启以后过两天又挂。经理让我看看怎么回事。说实话，第一次接到这种任务也有点慌，但硬着头皮捋了一遍，发现其实就三板斧。

## 调优三板斧：参数、工具、GC 日志

### 第一板斧：基础参数别搞错

很多面试从这里开始。最基础的那几个参数你得脱口而出：

```bash
-Xms4g          # 堆初始大小，通常设和 -Xmx 一样，避免扩容抖动
-Xmx4g          # 堆最大大小
-Xss512k        # 栈大小，太大浪费内存，太小容易 StackOverflow
-XX:MetaspaceSize=256m   # 元空间触发 Full GC 的阈值
-XX:MaxMetaspaceSize=256m
```

**个人经验：** 有次查一个问题，同事把 `-Xms` 设了 256m，`-Xmx` 设了 8g。启动后堆从 256m 慢慢涨，每次扩容都要触发一次 Full GC，系统前 10 分钟一直在 Full GC。改完 `-Xms8g -Xmx8g` 就稳了。这不是啥高深技术，就是把这两个值设成一样就行了。

面试时把这个例子说出来，比干巴巴背参数有用得多。

### 第二板斧：选对 GC 收集器

JDK 8 默认是 Parallel Scavenge + Parallel Old。JDK 11+ 默认 G1。

面试高频问题：**什么场景选什么 GC？**

| 收集器 | 适合场景 | 目标 |
|--------|---------|------|
| Serial GC | 客户端、单核小内存 | 低暂停并不重要 |
| Parallel GC | 批量处理、数据分析 | 吞吐量优先 |
| CMS | 老年代回收，追求低延迟（已废弃） | 低暂停 |
| G1 | 4g+ 堆，JDK 9+ 默认 | 平衡吞吐和延迟 |
| ZGC | 超大堆（几百 G），亚毫秒暂停 | 极低延迟 |

面试里八成问的是 G1，因为它是默认选择。G1 的核心参数就三个：

```
-XX:+UseG1GC
-XX:MaxGCPauseMillis=200   # 目标暂停时间，默认 200ms
-XX:G1HeapRegionSize=4m     # Region 大小
```

G1 最大的特点是**把堆分成多个 Region**，优先回收垃圾最多的 Region，所以叫 Garbage-First。

**我的个人经验：** 之前有个系统堆内存 32g，用的 G1，但每次 GC 暂停还是到了 500ms 以上。我一看配置，没有设 `-XX:MaxGCPauseMillis`，走的是默认值 200ms。但 Region 大小设太大，一个 Region 里面对象多，回收时间自然长。调小 Region 并显式设了暂停目标时间后，GC 暂停降到了 180ms 左右。面试时说到 G1，你可以把 Region 这个概念和调优方式讲清楚，一般面试官会点头。

### 第三板斧：看懂 GC 日志

会看 GC 日志，是调优入门和进阶的分水岭。

开启 GC 日志：

```bash
# JDK 8
-XX:+PrintGCDetails -XX:+PrintGCDateStamps -Xloggc:/path/to/gc.log

# JDK 11+
-Xlog:gc*:file=/path/to/gc.log:time,tags
```

日志里我最关注几个点：

**1. Young GC 频率。** 如果每 1-2 秒就一次 Young GC，说明 Eden 区太小，或者对象创建太猛。可以调大新生代或者检查代码是否有不必要的对象创建。

**2. Full GC 次数。** 正常的 Java 应用 Full GC 不应该频繁出现。一天几次算正常，一小时几十次就是有问题。

**3. GC 暂停时间。** 年轻代 GC 一般在几十毫秒内完成。如果超过 100ms 还持续好多次，需要查一下是不是对象晋升太快，或者老年代碎片化严重。

**常见 Full GC 原因：**

| 现象 | 可能原因 | 对策 |
|------|---------|------|
| Full GC 频繁但老年代才用到 30% | Metaspace 触发 | 调大 `-XX:MetaspaceSize` |
| Full GC 后老年代占用很快又涨满 | 内存泄漏 | 用 MAT 或者 jprofile 分析堆转储 |
| CMS 出现 Concurrent Mode Failure | 老年代回收速度跟不上分配速度 | 调大老年代或调整 CMS 触发阈值 |
| G1 频繁 Full GC | Humongous Allocation（大对象） | 增加 Region 大小或优化大对象<br>检查代码是否有超大数组/缓存 |
| Full GC 越来越频繁 | 代码里的对象没有被及时回收 | 检查集合类缓存是否一直增长 |

## 面试官还可能问什么

**"有没有做过 JVM 调优案例？"**

这个问题其实是给你舞台。别只说参数，说过程。

我当时是这么答的：

> 有。之前一个活动推送服务，用户量突然暴涨，系统频繁 Full GC。我先用 `jstat -gcutil` 确认 GC 频率，发现 Full GC 每 5 分钟一次。然后用 `jmap -histo:live` 看堆里什么对象最多，发现一个缓存 Map 里存了大量用户 Session，但只有 put 没有 remove。fix 了缓存逻辑后，Full GC 降到一天两三次。同时配合把堆从 4g 扩到 8g，GC 暂停也明显降下来了。

这个回答好在哪？有工具、有分析过程、有实际改动、有前后对比。

**"JVM 调优有什么原则？"**

记住一句话：**能不调的，尽量别调。**

默认参数经过了大量测试，大部分场景够用。乱调参数反而容易翻车。我是吃过亏的——刚开始学调优的时候在一个小项目上调了一堆参数，结果搞得 GC 更频繁了，改回来才恢复正常。

**"线上 Full GC 了，你怎么排查？"**

标准流程：

1. `jps` 找到进程 ID
2. `jstat -gcutil <pid> <interval>` 看 GC 频率和内存变化
3. `jmap -dump:format=b,file=heap.hprof <pid>` 导出堆转储
4. 用 MAT 或 JProfiler 分析大对象和 GC Root
5. 定位到泄漏点，修代码

## 面试能拿分的几个点

1. **说清楚 -Xms 和 -Xmx 设置成一样的原因**——不是为了装，是避免启动后频繁扩容 GC
2. **G1 的 Region 概念讲清楚**——知道 G1 不止是"大堆用 G1"
3. **带工具名称**——`jstat`、`jmap`、`jstack`、MAT，随口就能说出来
4. **说一个自己的案例**——哪怕是朋友项目的经历，也比背书强
5. **知道什么时候不调**——最高级的调优不是参数调得花里胡哨，是知道保持默认
