聊到 JVM 垃圾回收，大部分面试官都会问一句："你了解 JVM 的垃圾回收机制吗？"

这个问题范围太广了。你要是从"对象怎么判定存活"开始讲，讲到 G1 的 Mixed GC，面试官可能听着听着就走了。

面试嘛，讲究的是**点到即止 + 关键点清晰**。这篇就按面试的节奏来，把垃圾回收这摊子事捋一遍——够用就行，不贪多。

## GC 到底在回收什么？

JVM 里 GC 回收的对象都是**堆里的对象实例**。

对象什么时候该回收？没人引用它了。怎么判断没人引用？两种经典算法：**引用计数法**（Python 在用，JVM 不用，因为解决不了循环引用）和**可达性分析**（JVM 用的）。

可达性分析很简单：从 GC Roots 出发，一路找引用链。找不到的对象就是不可达的，标记为可回收。

GC Roots 包括什么？
- 虚拟机栈（栈帧中的局部变量表）里引用的对象
- 方法区中静态属性引用的对象
- 方法区中常量引用的对象
- 本地方法栈中 JNI 引用的对象

**个人经验**：排查内存泄漏的时候，用 `jmap -dump` 导 dump 文件，分析 GC Roots 的引用链，就能找到谁长期持有该释放的对象。这招排查线上泄漏比盲猜快十倍。

## 分代收集理论

JVM 把堆分成几块区域，每块用不同的回收策略：

- **新生代（Young Gen）**：存放"朝生夕死"的对象。占堆约 1/3，内部又分 Eden、From Survivor、To Survivor（默认比例 8:1:1）。
- **老年代（Old Gen）**：存放活的久的大对象或者多次 GC 没死掉的对象。
- **元空间（Metaspace，JDK 8+）**：替代永久代存放类元数据，用的是本地内存，不在堆里。

分代收集的依据是一句话：**绝大多数对象都活不过第一轮 GC**。

## 从 Serial 到 G1——面试够用版

### Serial GC

单线程收集器。它工作的时候，其他线程全部暂停（Stop The World，简称 STW）。

- 新生代：**标记-复制**算法
- 老年代：**标记-整理**算法

优点：简单高效，单核环境下比其他收集器好。
缺点：STW 时间太长。

你在用 `java -jar` 跑的小项目，默认就是 Serial（如果是客户端模式）。**基本上就是学习玩具和微服务单机测试的命。**

### ParNew

Serial 的多线程版。利用多核 CPU 并行做新生代回收，STW 还是有的，只是时间变短了。

JDK 8 的 Server 模式下，ParNew + CMS 是经典组合。很多老项目到今天还在用这套。

### Parallel Scavenge + Parallel Old

JDK 8 默认的组合。和 ParNew 类似，也是并行收集，但重点不同：

- ParNew 追求的是**STW 时间越短越好**
- Parallel 追求的是**吞吐量最大化**

吞吐量 = 运行用户代码的时间 / （运行用户代码的时间 + GC 时间）。比如跑了 99 分钟，GC 花了 1 分钟，吞吐量就是 99%。

**怎么控制？** 用 `-XX:MaxGCPauseMillis` 和 `-XX:GCTimeRatio`。

**个人经验**：我之前负责的一个大数据批处理服务，用的就是 Parallel Scavenge。因为是批量计算任务，吞吐量优先比低延迟更合理。调完 GCTimeRatio 之后，任务总耗时从 45 分钟降到了 38 分钟——这就是吞吐量思维的实际收益。

### CMS（Concurrent Mark-Sweep）

追求**低停顿**。核心思路是让 GC 线程和用户线程**同时**跑。

流程四步：
1. 初始标记（STW 极短）：标记 GC Roots 直接关联的对象
2. 并发标记（和用户线程一起跑）：从 GC Roots 往下追踪
3. 重新标记（STW 短）：修正并发标记期间变动的对象
4. 并发清除：清除垃圾对象

**问题**：
- 并发阶段吃 CPU，如果服务器负载本来就高，GC 可能会让性能雪上加霜
- 产生内存碎片（因为用的标记-清除算法）
- 浮动垃圾问题

JDK 9 开始标记 deprecated，JDK 14 正式移除。**还在用 CMS 的项目该迁移到 G1 了。**

### G1（Garbage First）

JDK 9 之后的默认收集器。设计思路完全不同：

- 不再分代的物理分区，把堆分成一个个 **Region**（1~32MB 的等大小区域）
- 新生代和老年代在逻辑上存在，物理上就是一些 Region 的集合
- 优先回收垃圾最多的 Region（所以叫 Garbage First）

**特点**：
- 可预测的停顿时间模型：用 `-XX:MaxGCPauseMillis` 设定目标停顿时间
- 没有内存碎片问题（用的是局部标记-复制）
- 不需要固定大小分区，自动调整新生代大小

G1 的大致流程（和 CMS 有点像）：
1. 初始标记（STW）
2. 并发标记
3. 最终标记（STW）
4. 筛选回收（STW，但可以选择性地回收）

**面试点**：面试官如果问你 G1 和 CMS 的区别，核心就两条——G1 解决了 CMS 的**内存碎片**和**停顿不可预测**两个问题。

## 面试高频题小抄

| 问题 | 一句话回答 |
|------|-----------|
| 怎么判断对象可回收？ | 可达性分析，从 GC Roots 出发找引用链 |
| 新生代 GC 怎么做 | 标记-复制：Eden + From 存活的对象复制到 To，Eden 和 From 清空，From 和 To 交换角色 |
| 为什么不用引用计数法？ | 解决不了循环引用 |
| CMS 和 G1 选哪个？ | G1。CMS 已移除，有碎片和停顿不可控问题 |
| G1 的优势？ | 可预测停顿、无碎片、自动调优 |
| 什么时候用 Parallel？ | 吞吐量优先的场景，比如后台批处理、大数据计算 |

## 一句话总结

面试 JVM GC，把可达性分析、分代收集、Serial→ParNew→Parallel→CMS→G1 这条线捋清楚，每个收集器说清楚"干什么、优缺点、STW 停在哪"，面试官大概率直接跳过这个话题——因为已经确认你懂了。
