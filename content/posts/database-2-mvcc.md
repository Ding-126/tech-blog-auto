面试问事务隔离级别，大部分人张口就来：RU→RC→RR→Serializable，RC 读已提交，RR 可重复读。然后呢？然后啥也说不出来了。

我之前做面试官时，发现一个规律：**能把四个级别名字背对的人很多，能说清楚 MVCC 为什么能做到非阻塞读的人，十个里不超过两个。**

这篇不跟你兜圈子——直接讲 MVCC 怎么工作的，RC 和 RR 到底差在哪，以及怎么跟面试官聊出层次感。

## 脏读、不可重复读、幻读——先对齐定义

聊隔离级别之前，得先对齐这仨异常场景。

我早期在一家创业公司写订单系统，遇到过这么个 Bug：A 事务读到 B 事务还没提交的中间数据（脏读），按那个数据做了发货操作。结果 B 事务回滚了，货已经发出去了。那就是典型的 RU 场景，MySQL 默认不会让你出这种问题，但某些场景（比如报表导出）确实有人故意把级别降到 RU 来提并发，**前提是你得知道自己在干什么**。

如果你不确定你的业务能不能接受脏读，**永远别用 RU。**

唯二需要记住的对比其实就两组：

| 场景 | 说明 | RC 能避免？ | RR 能避免？ |
|------|------|:---:|:---:|
| 脏读 | 读到别的事务未提交的数据 | ✅ | ✅ |
| 不可重复读 | 同一条记录两次读不一样 | ❌ | ✅ |
| 幻读 | 同一个范围两次查行数不一样 | ❌ | ✅（InnoDB 实际解决了） |

面试官问这个表，**你只需要解释清楚 RC 和 RR 就行了**——因为线上绝大多数系统就这两个选项。

## 别背名字，背原理

很多人理解隔离级别是按"能不能解决 XXX"去记的，表面上看没问题，换个问法就卡住了。

比如面试官问："RC 和 RR 在实现上有什么不同？"

答案不是"RC 比 RR 级别低"——那等于白说。应该回答：**它们底层都靠 MVCC（多版本并发控制），区别在于 snapshot 的创建时机。**

### MVCC 的核心：Read View

每条记录在 InnoDB 里不是只有一个版本——每次更新都生成一个新版本，老版本保留在 undo log 里。所以一条记录可能同时存在多个版本，像 Git 的历史提交一样。

MVCC 做的事很简单：**给每个事务分配一个 Read View（读视图），决定当前事务能看到哪些版本。**

Read View 里记录了一组事务 ID：

- m_low_limit_id：大于这个 ID 的事务不可见
- m_up_limit_id：小于这个 ID 的事务可见  
- m_ids：当前活跃的事务列表

判断一条记录对当前事务是否可见的规则是：

1. 如果记录的 trx_id 小于 m_up_limit_id → 已提交，可见
2. 如果记录的 trx_id 在 m_ids 里 → 未提交，不可见 → 沿着 undo log 找上一个版本
3. 如果记录的 trx_id 大于等于 m_low_limit_id → 未来事务，不可见

这套规则在 RC 和 RR 里**一模一样**。

区别在哪？**创建 Read View 的时机。**

### RC：每条语句都创建新的 Read View

在 RC 级别下，事务里的**每一条 SELECT 语句**都会重新创建一个 Read View。

所以事务 A 里第一次 SELECT 和第二次 SELECT 之间，如果别的事务提交了，第二次 SELECT 就能看到提交后的数据——这就是"不可重复读"的来源。

### RR：只用事务的第一个 Read View

在 RR 级别下，Read View **只在事务的第一条 SELECT 时创建一次**，后续全部复用。

不管别的事务提交了什么数据，当前事务始终只看到第一次创建 Read View 那个时刻的快照。所以同一条记录两次读结果一样，实现了"可重复读"。

**代价是什么？** undo log 不能被清理，因为老版本还需要被读取。长事务 = undo log 膨胀 = 磁盘空间爆炸。这是我踩过的坑——有个统计任务跑了 20 分钟没提交，导致前一天清理 undo log 的定时任务失效，磁盘报警。

## 幻读：RR 真的解决了吗？

教科书说 RR 解决不了幻读，InnoDB 的实际实现解决了——通过 **Next-Key Lock**。

Next-Key Lock 是 Record Lock + Gap Lock 的合体。不光锁住命中的记录，还锁住记录之间的间隙。

举个栗子：

```sql
SELECT * FROM user WHERE age > 20 FOR UPDATE;
```

在 RR 级别下，这条 SQL 不光锁住 age > 20 的所有行，还把 (20, +∞) 这个区间也锁了。别的事务想插入 age=25 的新记录？不行，gap lock 挡住了。

所以幻读在 InnoDB RR 级别下不会发生。

**但是**——如果用的是普通的快照读（不加锁的 SELECT），MVCC 的快照机制已经避免了幻读。

所以面试官问"RR 能解决幻读吗"，正确答案是：**快照读通过 MVCC 解决，当前读通过 Next-Key Lock 解决。**

这不是死记硬背，是理解了两套机制之后自然得出的结论。

## 面试怎么聊出层次感

**第一层**（及格）：能背出四个隔离级别和对应的异常场景。

**第二层**（良好）：能说清楚 RC 和 RR 在 MVCC 实现上的核心差异——Read View 创建时机。

**第三层**（优秀）：能讲出 RR 通过 Next-Key Lock 解决了幻读，并且知道什么时候 RR 没必要用、RC 更合适。

我曾经帮一个朋友改生产数据库的隔离级别，从 RR 降到 RC。原因很简单：业务场景不需要可重复读，但高并发下 RR 的 gap lock 导致大量插入等待，RT 从 5ms 飙到 200ms。改完后 RT 回落，数据完全没问题。

不是 RR 一定比 RC 好，是**看业务场景选级别**。

`RC + 行锁` 在某些高并发场景下远胜 `RR + 间隙锁`。

## 几个记住就行的小结论

- MVCC 是 InnoDB 的默认实现，不是可选的
- RC 和 RR 都靠 MVCC，区别是 Read View 创建时机
- 快照读（普通 SELECT）不加锁，当前读（SELECT ... FOR UPDATE / UPDATE / DELETE）才加锁
- RR 的 Next-Key Lock 能防幻读，但也可能成为性能瓶颈
- 长事务 = 性能杀手，能短则短
