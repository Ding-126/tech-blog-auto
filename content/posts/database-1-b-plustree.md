面试问索引，十次有八次会问到 B+Tree。背八股文的人都能说两句"B+Tree 是 B-Tree 的变体""叶子节点存数据"，但再深问一句"为什么 MySQL 选了 B+Tree 而不是别的"，很多人就卡住了。

今天这篇把这个事情说透。不讲虚的，直接扣数据结构。

## 面试官：MySQL 为什么用 B+Tree 而不是 B-Tree，也不是红黑树？

这个问题拆开其实是两个问题：

1. **为什么不用红黑树？**
2. **为什么从 B-Tree 升级到了 B+Tree？**

先说红黑树。没错，红黑树是平衡二叉树，查找是 O(log n)，听起来不错。但 MySQL 的数据是存在磁盘上的，每次读一个节点就是一次磁盘 IO。红黑树每个节点只存一个 key，树高随数据量线性增长。

举个例子：一张表 1000 万行数据，红黑树大概要 24 层。查一次走 24 次磁盘 IO，一次 IO 大概 10ms，240ms 就没了——这还是只查一次索引。聚簇索引要走两次，业务上的 SQL 动不动就回表，一来一回分钟级别。

有人问"可以加缓存啊"，缓存是内存层面的优化，跟数据结构没关系。树的高度决定了磁盘 IO 次数，这才是命门。

所以 MySQL 要的是一个**矮胖**的树。B-Tree 就是这个思路：一个节点存多个 key，扇出大，树就矮。

那为什么 Innodb 用了 B+Tree 而不是 B-Tree？区别在哪？

| 特性 | B-Tree | B+Tree |
|------|--------|--------|
| 数据存储 | 非叶子节点也存数据 | 只存索引，叶子节点存数据 |
| 叶子节点结构 | 独立 | 双向链表串联 |
| 范围查询 | 中序遍历，多次 IO | 走链表，顺序 IO |
| 非叶子节点大小 | 大（含数据） | 小（只存 key），扇出更大 |

**B+Tree 比 B-Tree 优势在哪？**

先说你最容易感知到的差别：**范围查询**。

B-Tree 做范围查询（比如 `WHERE id > 100 AND id < 200`），查到 100 之后得往回走，反复回溯，每个节点都可能涉及一次随机 IO。

B+Tree 呢？所有数据都在叶子节点，叶子节点之间用链表串起来了。找到 100 之后顺着链表往下走，顺序读就行。顺序 IO 比随机 IO 快一到两个数量级——这是 B+Tree 最核心的工程优势。

第二个优势很多人忽略：**扇出更大**。B+Tree 的非叶子节点只存 key，不存数据。MySQL 的页大小默认 16KB，一个 bigint 8 字节加指针 6 字节，一个节点能存大概 1170 个 key。三层 B+Tree 能存 1170 * 1170 * 16 ≈ 2000 万条记录。

三层 = 三次磁盘 IO，跟红黑树的 24 次比，差距就摆在这。

## 面试官：聚簇索引和非聚簇索引到底什么区别？

这个问题我最开始也背过答案："聚簇索引的叶子节点存整行数据，非聚簇索引存主键值"。面试基本分到手了。

但我想说的是，这个区别**不只是存储方式的不同**，它带来了几个很实际的后果。

**后果一：聚簇索引只能有一个，非聚簇索引可以有多个**

因为数据只有一份，按一个顺序排列。主键索引决定了数据在磁盘上的物理顺序，所以它只能有一个。非聚簇索引（也叫二级索引）的叶子节点存的是主键值，不存实际数据——所以可以有很多个。

**后果二：回表是个性能杀手**

用二级索引查询时，先到二级索引的 B+Tree 里找到主键值，再到聚簇索引的 B+Tree 里去查完整行。这就多了一次 B+Tree 的磁盘 IO。

所以 `SELECT *` 走二级索引，大概率要走回表。但如果你只查索引覆盖的列，就不用回表——这就是**覆盖索引**的原理。

我之前负责过一个报表接口，跑一次要 3 秒。排查发现查了 5 个字段，但索引只覆盖了 2 个，每次要回表查剩余 3 个字段。把那个 SQL 改成只查索引覆盖的列 + 再加一个联合索引覆盖剩余字段，耗时降到了 200ms。收益极其明显。

**后果三：主键设计决定了性能天花板**

既然聚簇索引决定了数据物理顺序，那主键怎么选就很重要了。

用自增 ID 做主键，新数据永远插在 B+Tree 的最右边，不会触发页分裂。用 UUID 做主键，每次插入都有可能落在中间某个位置，触发页分裂和页重组，写性能直接打折。

我见过一张表用 UUID 做主键，插入性能只有自增 ID 的 1/3。后来业务不能改主键类型，只能把自增 ID 作为隐藏主键，UUID 做逻辑主键，才算救回来。

## 面试官：最左前缀原则怎么理解？

联合索引（a, b, c）在 B+Tree 里怎么存的？

先按 a 排序，a 相同按 b 排序，b 相同按 c 排序。所以查询条件必须从最左边开始匹配，才能利用这个索引。

`WHERE a=1 AND b=2` ✅ 能走索引
`WHERE b=2` ❌ 不能走——跳过了 a，相当于直接查第二列
`WHERE a=1 AND c=3` ⚠️ 只能用到 a，c 用不上——跳过了 b

这个问题很多面经里都有，但我自己的经验是：**联合索引字段的顺序，要按照查询频率和区分度来排**。区分度高的放前面，最常用的查询条件放前面。

## 总结

做面试准备也好，写业务代码也好，理解 B+Tree 的核心就够了：

1. **矮胖**——用更少的磁盘 IO 查到数据
2. **叶子链表**——范围查询走顺序 IO
3. **聚簇索引存数据**——理解了这个，回表、主键设计、覆盖索引就都通了

下次面试官问你 B+Tree，别只背特点。把"为什么不用红黑树""为什么 B+Tree 比 B-Tree 好""聚簇和非聚簇到底有什么实际影响"这几个点说清楚，面试官就知道你是真懂了。
