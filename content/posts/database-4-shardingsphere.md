分库分表是面试里逃不掉的话题。大厂面试官基本必问，中厂也在跟进。

你背了很多概念——垂直拆分、水平拆分、路由策略——但面试官一个问题就能把你问住："你们的业务量到了多少，你才决定分的？"

说不上来，对吧。

这很正常。分库分表最难的不是怎么分，而是**什么时候分**。今天这篇就从决策时机讲到落地工具，最后聊面试官真正想看什么。

## 什么时候该分？别被"百万数据"忽悠了

网上很多文章说"单表超过500万就要分"，纯属刻板印象。

我在前公司遇到过一个真实案例：一张日志表 2000 万行，查询只需要 `select count(*) from log where date = yesterday` 这种简单聚合，单表跑得飞起，完全不用分。反而是另一张只有 20 万行的订单表，因为业务方在上面建了 15 个索引，写一次要维护一堆索引树，慢得不行。

**分库分表的决策依据不是数据量，是瓶颈类型。**

我从实际踩坑里总结了三类"必须分"的场景：

**1. 单库连接数打满**

这是最常见的"不分布不行"的场景。你的应用有几十个微服务，每个服务连同一个库，连接池一开就是几百个。库的连接数上限是固定的（MySQL 默认 151），到了阈值，新请求直接排队。这个时候你加索引、换硬件都没用，因为瓶颈在**连接**而不是查询速度。

**2. 单表写入撑不住**

典型特征：主从延迟持续走高，从库追不上主库的写入速度。这个时候加只读实例已经没意义了——不是读不过来，是写不过来。只有把写流量分散到多个库，才能解。

**3. 单表数据量导致索引效率崩了**

B+ 树的深度在数据量超过千万级别时从 3 层变成 4 层。每次查询多一次 IO，看起来不多，但如果是高频查询，累积的 RT 增长就明显了。不过这个阈值取决于你的行宽度和索引大小，不是固定数字。我一般用 `show table status` 看 `Data_length` + `Index_length`，超过 10GB 就开始关注。

## 垂直分还是水平分？面试官想听你的判断

面试里谈到分库分表，面试官大概率会追问："你们用了哪种方式？"

两种方式我都会。但真正让面试官点头的，是你**说了为什么选它**。

**垂直拆分（按业务拆分）**：把用户表、订单表、商品表分到不同的库。这是微服务架构的自然延伸——订单服务只管订单库，商品服务只管商品库。

优点：解耦干净，每个库独立维护。
缺点：跨库查询让你想骂人。一个简单的 JOIN 没了，你得在应用层做数据拼接，或者忍受多次网络 IO。

**水平拆分（按数据分片）**：把同一张表按某个字段（通常是用户 ID 或订单 ID）拆分到多个库或多张表。这是解决"单表太大"的标准方案。

我做过一个项目，订单表每天增长 50 万行，三个月就上亿了。我们按 `order_id` 的哈希分到了 8 个库，每个库 16 张表（总共 128 张物理表）。效果立竿见影，写入速度从 2000 tps 涨到了 12000 tps。

但这不是没有代价的。代价后面说。

**面试技巧：** 不要只说"我们用了水平拆分"。要说清楚——因为当时的瓶颈是什么，为什么垂直拆分不能解决问题，水平拆分解决了什么问题，带来了什么新问题。

## ShardingSphere 的核心概念

聊完什么时候分、分什么，接下来就是怎么分。

现在业界做分库分表，ShardingSphere 几乎是标准答案。面试问你分库分表，十有八九会绕到它。

ShardingSphere 的核心就一句话：**把你的 SQL 改写成能在多个库上执行的 SQL，再把结果合并返回。**

它分三个产品：

- **ShardingSphere-JDBC**：轻量级 Java 框架，在 JDBC 层做拦截。适合单体应用，不需要额外部署。你的项目里加个 jar 包，配几个规则就能用。
- **ShardingSphere-Proxy**：独立部署的代理层，对应用透明。适合异构系统——你只要把数据库连接地址指向它，不需要改代码。
- **ShardingSphere-Sidecar**：Service Mesh 模式，目前用得还不多，面试一般不问。

面试高频考点是 **ShardingSphere-JDBC**。

它的工作原理像一层"SQL 中间人"：

```
你的应用 → ShardingSphere-JDBC → 多个数据库
           (解析 SQL → 路由 → 改写 → 执行 → 结果归并)
```

具体每一步：

1. **SQL 解析**：把 SQL 解析成抽象语法树（AST），找到需要分片的字段和值
2. **路由**：根据分片键，计算这条 SQL 应该去哪个库的哪个表
3. **SQL 改写**：把逻辑表名替换成物理表名，比如 `t_order` → `t_order_0003`
4. **执行**：在目标库上执行改写后的 SQL
5. **结果归并**：把多个库的返回结果拼在一起，排序、分页、聚合一次性搞定

## 配置实战：一个订单表怎么拆

我拿订单表举个实际例子。

假设我们有订单表 `t_order`，要按 `user_id` 分片。配置文件长这样（ShardingSphere-JDBC 的 YAML 配置，面试手写也不难）：

```yaml
rules:
- !SHARDING
  tables:
    t_order:
      actualDataNodes: ds$->{0..1}.t_order_$->{0..15}
      tableStrategy:
        standard:
          shardingColumn: user_id
          shardingAlgorithmName: t_order_inline
      keyGenerateStrategy:
        column: order_id
        keyGeneratorName: snowflake

  shardingAlgorithms:
    t_order_inline:
      type: INLINE
      props:
        algorithm-expression: t_order_$->{user_id % 16}

  defaultDatabaseStrategy:
    standard:
      shardingColumn: user_id
      shardingAlgorithmName: database_inline

  databaseAlgorithms:
    database_inline:
      type: INLINE
      props:
        algorithm-expression: ds$->{user_id % 2}
```

解释一下这几行在做什么：

- `actualDataNodes`：告诉 ShardingSphere，物理库 `ds0` 和 `ds1` 里分别有 `t_order_0` 到 `t_order_15` 共 16 张物理表
- 路由规则：`user_id % 16` 决定去哪张表，`user_id % 2` 决定去哪个库
- 主键用雪花算法（Snowflake）生成，保证全局唯一且递增

一个具体的用户请求来查 `user_id = 10086`，ShardingSphere 算出来它应该去 `ds0.t_order_6`，只查那一张表——效率极高。

## 分库分表后的三个坑

面试进阶题：分库分表带来了什么新问题？

**第一个，跨分片查询。**

你的 `user_id` 作为分片键看起来很好用，但突然有一天产品说："帮我查一下上周所有状态为 '待支付' 的订单。"

你没法用 `user_id` 查了——因为不知道哪些用户下了待支付订单。你只能把所有 128 张表全扫一遍，然后归并结果。慢不慢？慢。但这就是做分片查询的代价。

**兜底方案**：建一张索引表，存 `（状态, 分片键）` 的映射关系。或者用 ES 做二级索引。

**第二个，分布式事务。**

分库前，一个 `insert into t_order` + `update t_inventory` 在一个事务里搞定。分库后，订单在 `ds0`，库存可能在 `ds1`，本地事务管不了跨库的。

常见方案：基于消息队列的最终一致性（适合大部分业务场景），如果需要强一致性就用 Seata 的 AT 模式。

**第三个，分页和排序。**

`ORDER BY create_time DESC LIMIT 10` 在单表上是精确的，但分库后——每个库各自返回自己 TOP 10，ShardingSphere 拿到 20 条再排一次取 TOP 10，结果是不准确的吗？

**不一定。** 对于大多数业务场景（比如按时间倒序看最近的 10 条订单），由于 `create_time` 趋势上是单调递增的，最新数据大概率集中在一两个库，误差可以接受。但对于精准翻页（比如用户想看第 5 页的数据），必须考虑合并排序的准确性，ShardingSphere 的归并引擎做了这个事情，但跨多页查询的性能会急剧下降。

**面试技巧：** 能主动说出这些坑，说明你真的干过，不是背书的。

## 写在最后

分库分表不是银弹。很多人还没遇到瓶颈就急着拆，结果系统复杂度上去了，性能没提升多少。

正确的做法是：**不到万不得已，不分。** 等到确认真的是数据库瓶颈了，先试试缓存、读写分离、冷热分离这些轻量方案，都试过了还不行，再上分库分表。

我带过的一个团队，前期花了大量精力在 "要不要分" 的论证上。最后决定分，但分了之后又花了更多精力处理分片查询和分布式事务。所以还是那句话：**分库分表解决了一个大问题，但带来了三个新问题。确保你真正需要解决那个大问题，再来。**
