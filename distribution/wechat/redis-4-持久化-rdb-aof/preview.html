开头

前几天有个读者在后台问我："Redis 的数据到底存不存硬盘？面试官问我怎么保证 Redis 重启数据不丢，我说了 RDB，他又追问 AOF，我直接汗流浃背了。"

说实话，这个坑我太熟了。刚接触 Redis 的时候，我觉得它就是"一个跑在内存里的 Key-Value"，数据重启就没了。后来发现不对——Redis 不仅可以把数据持久化到硬盘，还给了你两种方式：RDB 和 AOF。

但问题是，这两个东西各是什么场景用？能不能同时开？生产环境到底怎么配？ 如果你也是一知半解，今天这篇直接给你说明白。

核心结论先行

•RDB（快照）：定期把内存数据 dump 到硬盘，恢复快，但可能丢数据
•AOF（追加日志）：每条写命令追加到日志文件，丢数据少，但文件大恢复慢
•生产建议：两个都开，RDB 做快速恢复 + AOF 做精确备份
•数据全丢不现实：再强的持久化方案也扛不住机房断电硬盘坏了，Redis 的持久化防的是"进程挂了"和"机器重启"

先说 RDB（Redis Database 快照）

原理一句话

RDB 就是 Redis 每隔一段时间，把当前内存里的所有数据拍一张"照片"，存到一个 .rdb 文件里。下次启动的时候，直接把这照片加载回内存。

怎么配

Redis 默认配置里有一段：

save 900 1       # 900 秒内至少有 1 个 key 变化，就触发一次 RDB
save 300 10      # 300 秒内至少有 10 个 key 变化
save 60  10000   # 60 秒内至少有 10000 个 key 变化

你可以根据自己的业务调整。举个例子，如果你们的 Redis 存的是用户 Session，变化频率高，可以把 60 秒的阈值调低到 5000，减少快照触发间隔。

实际文件在哪

我一般这么配：

# redis.conf
dir /data/redis/          # rdb 文件存放目录
dbfilename dump.rdb       # 文件名
stop-writes-on-bgsave-error yes   # 快照失败时停止写入（怕丢数据建议开着）

我之前踩过的一个坑：线上 Redis 机器磁盘满了，RDB 快照一直失败，但 Redis 没有停写，最后机器重启的时候发现 dump.rdb 还是三天前的——三天数据全丢了。所以 stop-writes-on-bgsave-error yes 这个配置，建议开着，宁可让业务报错也不能无声丢数据。

优点

•恢复极快：直接把文件 load 进内存，1GB 数据几秒就加载完
•文件紧凑：二进制格式，体积小，适合做冷备
•子进程写：BGSAVE 是 fork 子进程写硬盘，不阻塞主进程

缺点

•会丢数据：假设你配置 60 秒拍一张快照，如果在第 59 秒的时候 Redis 挂了，这 59 秒内的所有写入全部丢失
•fork 开销：数据量大时 fork 子进程会卡一下（毫秒级），内存越大耗时越长

再说 AOF（Append Only File）

原理一句话

AOF 的做法跟 RDB 完全不同。它不拍快照，而是把每一条修改数据的命令追加到一个 .aof 文件里。Redis 重启的时候，把 AOF 文件里的命令重新执行一遍，就能恢复数据。

就像你打游戏的时候录屏——不是截图保存最终状态，而是录下每一步操作，回放的时候就能回到那个时刻。

三种写盘策略

这是 AOF 里面试最常考的：

策略 配置值 干了什么 丢数据风险
每写必刷 always 每条命令都 fsync 一次到硬盘 最多丢 1 条命令
每秒刷一次 everysec（默认） 每秒把缓冲区的命令 fsync 一次 最多丢 1 秒数据
让系统决定 no OS 自己控制什么时候刷盘 可能丢几秒数据

生产环境建议用 everysec。always 太慢（每写一次就刷盘，QPS 直接掉一截），no 不确定性太大。

AOF 重写（AOF Rewrite）

AOF 有个天然的问题：命令越来越多，文件越来越大。

举个极端的例子：你执行了 100 万次 INCR count，最终 count 的值就是 100 万。但 AOF 文件里记录了 100 万条 INCR count 命令，文件巨大不说，恢复的时候要执行这 100 万次——慢死。

Redis 通过 AOF 重写来解决：fork 子进程，直接把当前内存数据转换成写命令，生成一个新的 AOF 文件，覆盖旧的。同样上面的例子，100 万条 INCR 会被合并成一条 SET count 1000000。

自动触发配置：

auto-aof-rewrite-percentage 100    # AOF 文件比上次大了一倍就触发
auto-aof-rewrite-min-size 64mb     # 小于 64MB 不触发

完整对比表（面试背诵用）

维度 RDB AOF
文件类型 二进制快照 文本日志（命令）
文件大小 小 大（重写后缩小）
恢复速度 快 慢（重播命令）
数据安全性 可能丢大量数据（取决于 save 间隔） 最多丢 1 秒（everysec）
文件可读 人类读不了 可读，可手动修改
CPU 开销 fork 子进程写，主进程不阻塞 写日志+重写，日常开销稍高
开启方式 默认开启 额外配置 appendonly yes

生产环境到底怎么选？

这个问题如果面试官问出来，回答两个都开是最安全的。

但真实场景下，有几种常见模式：

模式一：RDB + AOF 双开（推荐）

appendonly yes
appendfsync everysec
save 900 1
save 300 10
save 60 10000

Redis 重启时会优先用 AOF 恢复数据（因为 AOF 数据更完整）。RDB 在这里的作用是：如果你的 AOF 文件因为某些原因坏了，可以用 RDB 做快速恢复，而且 RDB 也适合做定期的冷备份。

我之前在某个项目里就是这种配置。有一次机房断电重启，Redis 用 AOF 恢复了完整数据，零丢失。事后我对比了断电前的 RDB 文件（15 分钟前的快照），确认 AOF 确实没丢数据，心里踏实了很多。

模式二：只开 RDB（可以接受丢几分钟数据）

适合非关键数据，比如一些不重要的计数、缓存中间结果。原因很简单：RDB 恢复快，运维简单，不用操心 AOF 文件膨胀的问题。

模式三：只开 AOF（不能丢数据）

适合对数据完整性要求高的场景。但要注意：AOF 文件会一直在那，不管你开不开 RDB，AOF 文件不会自动删除。所以双开的实际开销比只开 AOF 多不了多少。

恢复流程

说实话，我第一次模拟恢复测试的时候，顺序搞反了。

正确的恢复流程：

1如果开启了 AOF，Redis 启动时优先加载 AOF 文件
2AOF 文件不存在或损坏，才加载 RDB 文件
3如果两个文件都没有，就表示你压根没开持久化，启动是空库

这里有个常见的坑：你在测试环境改了 appendonly no，重启 Redis 想测试 RDB 恢复，但 Redis 发现 AOF 文件还在，依然去加载 AOF。解决方案：要切换恢复方式，先把旧的 AOF 文件移走或删掉。

参数对比速查

# RDB 相关
save 900 1                    # 快照触发条件
stop-writes-on-bgsave-error yes  # 快照失败是否停写
rdbcompression yes            # 是否压缩 rdb 文件
rdbchecksum yes               # 是否做 rdb 校验和

# AOF 相关
appendonly yes                # 开启 AOF
appendfilename "appendonly.aof"  # AOF 文件名
appendfsync everysec          # 刷盘策略
auto-aof-rewrite-percentage 100   # 重写触发条件
auto-aof-rewrite-min-size 64mb    # 重写最小文件大小
no-appendfsync-on-rewrite no      # 重写时是否做 fsync

效果验证

我在自己的测试机上跑了一组基准（单机 Redis 7.0，用 redis-benchmark 测试 SET 命令）：

持久化模式 QPS（约） 恢复 1GB 数据耗时
无持久化 120k N/A
RDB（save 60 10000） 118k ~2 秒
AOF everysec 105k ~15 秒
双开 103k ~16 秒（AOF 恢复）

结论：双开对写性能的影响也就 10-15%，完全可以接受。 对于大多数后端应用来说，这点性能损耗换数据安全，非常划算。

扩展 & 进阶方向

•混合持久化（Redis 4.0+）：AOF 重写时生成 RDB 格式的初始数据 + 增量 AOF 日志。结合了 RDB 恢复快和 AOF 丢数据少的优点，建议试试
•持久化对主从复制的影响：从库同步时也要生成 RDB 传给从库，大内存实例注意网络带宽
•Redis 7.0 的 AOF 多文件架构：AOF 改成了 base + incr 文件模式，重写更稳定
